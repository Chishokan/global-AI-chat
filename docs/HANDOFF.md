# HANDOFF — 特定技能サポートアプリ / 機能② 多言語Q&Aチャット

Claude Code 向け引き継ぎドキュメント。CLAUDE.md の詳細版。最初にここを読めば、何を・なぜ・どう作るかが分かるようにしてある。

## 1. 背景・目的

NEP（学習塾グループの関連会社）が立ち上げる、特定技能・留学生として日本で働く外国人材向けの統合サポートアプリ。狙いは、来日後の「学習」「生活」「行政手続き」のつまずきをアプリで支え、長期安定就労と試験合格を後押しすること。

現在 NEP は日本語試験 **JOT（日本語オンラインテスト／CBT）** を導入・営業中で、本アプリはその周辺エコシステム（学習→受検→就労→定着）を構成する位置づけ。

構想全体は5機能（①AI学習 ②多言語チャット ③生活コンテンツ ④手続きナビ ⑤コミュニティ）。本リポジトリではまず **②** を作る。

## 2. これまでの検討と成果物（参考）

- 動くプロトタイプを React 単一コンポーネントで作成（チャットUI＋Claude API＋KBグラウンディング＋同言語応答＋やさしい日本語＋スタッフ誘導）。**本実装の挙動仕様の参照元**。
- GAS（Apps Script）版の下書きも作成済みだが、**本番は git ベースで作り直す方針**。GAS は使わない。ロジック（プロンプト、ログ列、レポート2層構成）は流用する。
- FAQ知識ベース（30項目）を作成。→ /data/faq.json が正本。

## 3. スコープ

### 作る（MVP）

- 多言語Q&Aチャット（KBグラウンディング・同言語応答・やさしい日本語・スタッフ誘導の座席）
- 会話ログの保存（Phase 2）
- 日次のAIレポート（①運営サマリー ②要フォロー抽出）を職員向けに出力（Phase 2）

### 作らない（後フェーズ）

- **機能① AI学習**: 既存eラーニングとの棲み分けが未確定。方針合意後に着手。
- **メンタル深刻検知のリアルタイム自動介入**: MVPは「スタッフ誘導の座席」＋「日次レポートでの要フォロー抽出」まで。
- **LINEチャネル配信**: まずWebで作る。チャット応答ロジックはチャネル非依存に切り出し（`lib/chat.ts`）、後でLINE Messaging APIに差せる設計。

## 4. 機能要件

### 4.1 チャット（Phase 1・実装済み）

- 利用者がテキストで質問→Claudeが **FAQ知識ベースに基づいて** 回答。
- **同言語応答**: 入力言語と同じ言語で返す。KB正本は日本語で持ち、出力時にLLMが変換。
- **やさしい日本語トグル**: ONのとき日本語回答を平易化。
- **KB外の扱い**: 該当情報が無ければ断定せず「スタッフに確認を」へ誘導。地域・個人で変わる情報は「市の最新案内で確認」を添える。
- **在留資格/ビザの法的判断はしない**: 専門窓口/スタッフへ。
- **スタッフ誘導**: 「スタッフに相談」操作で人につなぐ座席（MVPでは記録＋運用窓口提示）。
- モバイル前提。要点を先に短く、詳細（連絡先等）は後に。

### 4.2 会話ログ（Phase 2）

- 1往復ごとに保存。少なくとも: 日時・セッションID・言語/やさしい日本語フラグ・カテゴリ・質問・回答。
- **氏名等の直接識別子は保存しない**（セッションIDで扱う）。
- カテゴリは自動分類。

### 4.3 日次AIレポート（職員向け・Phase 2）

2層構成：

- **運営サマリー**: 件数、カテゴリ内訳、よく出た質問/ニーズ、スタッフ誘導になった話題。
- **要フォロー抽出**: メンタル不調・トラブル・困窮・緊急のサインを level(高/中/低) と type(メンタル/トラブル/困窮/緊急/その他) と理由つきで提示。保守的に拾い、最終判断は人。
- 実行: Vercel Cron で日次（例: 23:00 JST）。前回実行〜今回までを対象。

## 5. アーキテクチャ

- **Next.js (App Router) + TypeScript**。チャットUIはクライアント（`app/page.tsx`）、Claude呼び出しは route handler（`app/api/chat/route.ts`）。
- 応答ロジックはチャネル非依存に `lib/chat.ts` に切り出し（`generateReply`）。
- **LLM**: Anthropic Claude API（`lib/claude.ts`）。チャット応答は応答性重視のモデル。モデルIDは環境変数 `ANTHROPIC_CHAT_MODEL` で上書き可（既定: `claude-sonnet-4-6`、最新はコンソールで確認）。
- データフロー: 利用者 → /api/chat →（KB注入→Claude→応答）。Phase 2 でログ保存とカテゴリ分類を本ハンドラに追加。
- Phase 2: **Prisma + PostgreSQL**、`/app/api/cron/report`（`CRON_SECRET` で保護）。

## 6. データモデル（Phase 2・Prisma 目安）

Faq / Conversation / Message / Report / FollowUp の5モデル。氏名は保存せず sessionId で扱う。
（Phase 2 着手時に `prisma/schema.prisma` を作成する。設計の詳細は元HANDOFFのデータモデル節を参照。）

## 7. プロンプト

`lib/prompts.ts` に集約。`{KB}` はFAQから組み立てた知識ベース文字列（`lib/kb.ts` の `buildKnowledgeBase`）、`{EASY_JP}` はトグルON時のみ差し込む。

- **チャット用システムプロンプト**: KBグラウンディング・正確性ルール・在留資格の法的判断回避・同言語応答・やさしい日本語・トーンを規定。
- **レポート解析用プロンプト（Phase 2）**: JSONのみを返させ（コードフェンス除去）、summary / byCategory / topNeeds / unanswered / followUps を出力。診断や断定はせず、保守的にフォロー候補を拾う。

## 8. 知識ベース運用

- 正本: /data/faq.json（30項目）。Google Sheets「FAQ」シートが人間の編集元 → リポジトリへエクスポートして版管理。
- Phase 2 でDBへシード。アプリはDB（またはfaq.json）からKB文字列を構築。
- status: 要確認 の3項目（ハラル対応店/礼拝所・JOTの特定技能/育成就労扱い・家族呼び寄せ）は内容が変わりやすい。公開・更新時に再確認。
- 各窓口の電話・住所・受付時間は変わり得るため、公開前に公式ページで再確認するフローを運用に含める。

## 9. 非機能・ガードレール

- **セキュリティ**: APIキー・DB資格情報はサーバー側／環境変数。クライアントへ出さない。Cronエンドポイント（Phase 2）は CRON_SECRET で保護。
- **プライバシー**: 会話のAI解析・職員共有は、利用規約と初回オンボーディングで本人に明示する前提。レポート/ログは氏名でなくセッションID。
- **安全**: レポートは職員の気づき補助であり診断ではない。緊急性は日次バッチに依存させない。チャットは常に緊急番号・相談窓口（119/110/118、FRESC等）を案内できる状態を維持。
- **多言語/アクセシビリティ**: UIの可読性・やさしい日本語・モバイル最適化を重視。
- **チャネル非依存**: 応答ロジックをチャネルから切り離し、後でLINEに載せられるようにする。

## 10. デプロイ

- git push → Vercel 自動デプロイ。
- 環境変数を Vercel に設定（`ANTHROPIC_API_KEY` ほか）。
- Phase 2: マイグレーションを CI かデプロイフックで実行。Vercel Cron で /app/api/cron/report を日次起動。
